# ダウンロード自動振り分けアプリ 仕様書

## 概要

`~/Downloads` フォルダに追加されたファイルを検知し、ユーザーにプロジェクト選択を促して、指定フォルダ内の日付フォルダへ自動移動するmacOS向けデスクトップアプリ。

---

## ディレクトリ構成

```
.
├── main.py                 # エントリーポイント
├── projects.json           # プロジェクト設定ファイル
├── settings.json           # アプリ設定ファイル（監視フォルダなど、自動生成）
├── move_log.csv            # 移動ログ（自動生成）
├── setup.sh                # セットアップスクリプト
├── docs/
│   └── spec.md             # 本仕様書
└── src/
    ├── __init__.py
    ├── config.py           # 定数・パス設定・アプリ設定の読み書き
    ├── project_manager.py  # projects.json の読み書き
    ├── file_mover.py       # ファイル移動 & ログ記録
    ├── ui.py               # Tkinter ポップアップUI
    ├── settings_ui.py      # 設定画面UI
    └── watcher.py          # watchdog によるフォルダ監視
```

## 機能仕様

### 1. フォルダ監視

| 項目 | 内容 |
|------|------|
| 監視対象 | `settings.json` の `watch_folder`（未設定時は `~/Downloads`） |
| 監視方式 | watchdog（`FileSystemEventHandler`） |
| 検知イベント | `on_created`（新規作成）、`on_moved`（リネーム） |
| 除外対象 | `.crdownload` `.part` `.tmp` `.download` `.partial` |
| 重複防止 | `_pending` セットで処理中ファイルを管理 |
| 遅延処理 | ファイル書き込み完了を待つため0.8秒の遅延あり |
| 起動時検証 | 監視フォルダが存在しない場合はエラーを表示して終了 |

### 2. UI（ポップアップ）

- 新規ファイル検知時に Tkinter ウィンドウを表示
- 常に最前面に表示（`-topmost`）
- 画面中央に配置

表示内容：

| 要素 | 内容 |
|------|------|
| ファイル名 | 検知したファイルのベース名 |
| プロジェクト一覧 | `projects.json` から読み込んだ一覧（Listbox） |
| 振り分けるボタン | 選択プロジェクトへ移動 |
| 振り分けしないボタン | Downloads に残してログ記録 |
| ⚙ 設定ボタン | 設定画面を開く（閉じると一覧を再読み込み） |
| パス不明表示 | 登録済みプロジェクトのフォルダが存在しない場合「⚠ 名前（パス不明）」と表示 |

### 2-1. 設定画面（`src/settings_ui.py`）

`python main.py --settings` または振り分けダイアログの「⚙ 設定」ボタンから起動。

| 機能 | 内容 |
|------|------|
| プロジェクト一覧表示 | 名前・パスをテーブルで表示 |
| 追加 / 更新 | 名前・パスを入力して保存 |
| フォルダ参照 | ダイアログでフォルダを選択（名前を自動補完） |
| 削除 | 選択行を削除（確認ダイアログあり） |
| 即時反映 | 保存後、振り分けダイアログの一覧に自動反映 |
| 監視フォルダ変更 | 設定画面下部から監視フォルダを変更・保存（次回起動から反映） |

### 3. プロジェクト管理

- `projects.json` で管理
- フォーマット：

```json
{
  "プロジェクト名": "/絶対パス/プロジェクトフォルダ"
}
```

- 起動時に読み込み、UIに一覧表示
- プロジェクトフォルダが存在しない場合はエラーダイアログを表示
- `projects.json` が破損（不正JSON）の場合は自動バックアップ（`.bak_YYYYMMDD_HHMMSS`）を作成し、空の辞書として扱う
- `save_projects()` はアトミック書き込み（`.tmp` 経由で `os.replace`）で途中破損を防ぐ

### 3-1. アプリ設定（`settings.json`）

- `settings.json` でアプリ全体の設定を管理（プロジェクトとは分離）
- フォーマット：

```json
{
  "watch_folder": "/絶対パス/監視フォルダ"
}
```

- 未設定・ファイル不在の場合は `~/Downloads` をデフォルト値として使用
- 設定画面の「監視フォルダ設定」から変更・保存が可能（次回起動から反映）

### 4. ファイル振り分け

- 移動先：`{プロジェクトフォルダ}/{YYYY-MM-DD}/{ファイル名}`
- 日付フォルダは存在しない場合に自動作成（`os.makedirs`）
- 同名ファイルが存在する場合：`{ファイル名}_{HHMMSS}.{拡張子}` にリネーム
- **移動失敗時のロールバック**：`shutil.move` が失敗した場合、このコードが作成した空の日付フォルダを自動削除し、余分なフォルダが残らないようにする

### 5. ログ

- ファイル：`move_log.csv`
- 形式：CSV

| カラム | 内容 |
|--------|------|
| datetime | 処理日時（YYYY-MM-DD HH:MM:SS） |
| filename | ファイル名 |
| destination | 移動先フルパス |
| project | プロジェクト名（振り分けなしの場合は「（振り分けなし）」） |

## セットアップ

### ワンクリックインストール（推奨）

`install.command` を右クリック →「開く」で実行。以下を自動処理する：

1. Python 3.10 以上の存在確認
2. tkinter の確認（未インストール時は Homebrew 経由で自動インストール）
3. 仮想環境 `.venv` の作成
4. `watchdog` のインストール
5. `kiro.command` への実行権限付与

### 手動インストール

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install watchdog
```

## 実行

### ワンクリック起動（推奨）

`kiro.command` をダブルクリック。

### ターミナルから起動

```bash
source .venv/bin/activate

# 通常起動（フォルダ監視）
python main.py

# 設定画面のみ起動
python main.py --settings
```

終了は `Ctrl+C`。

## OS・他アプリへの干渉確認

コード全体（`main.py` / `src/` 配下全モジュール）を静的解析した結果、以下の通り干渉がないことを確認済み（2026-04-05）。

### ネットワーク

- `socket` / `urllib` / `requests` / `http` 等のネットワーク関連モジュールのインポートなし
- 外部サーバーへの通信は一切行わない
- **結論: 完全ローカル動作、外部通信なし**

### 外部プロセス

- `subprocess` / `os.system` / `os.popen` / `multiprocessing` の使用なし
- スレッドは `threading.Thread(daemon=True)` のみ使用（メインプロセス終了と同時に自動消滅）
- **結論: 外部コマンド・プロセスの生成なし**

### ファイルシステムへのアクセス範囲

| 対象 | 操作 | タイミング |
|------|------|-----------|
| 監視フォルダ | イベント受信のみ（読み取りなし） | 常時 |
| 監視フォルダ内ファイル | `shutil.move`（移動元として削除） | ユーザーが「振り分ける」を押した時のみ |
| プロジェクトフォルダ | 日付サブフォルダ作成・ファイル移動先 | ユーザーが「振り分ける」を押した時のみ |
| `projects.json` / `settings.json` | 読み書き（アプリフォルダ内） | 設定変更時のみ |
| `move_log.csv` | 追記（アプリフォルダ内） | 振り分け・スキップ操作時のみ |
| `/System` / `/Library` / `~/.ssh` 等 | アクセスなし | — |

- **結論: ファイルアクセスはユーザーが設定したフォルダとアプリ自身の設定/ログファイルのみに限定**

### システムリソース（CPU・メモリ）

- フォルダ監視は macOS の `FSEvents` API を利用（watchdog 経由）。カーネルからのコールバック方式のため、ポーリングよりも低負荷
- メインループは `time.sleep(1)` で休眠。CPU 使用率はアイドル時ほぼ 0%
- ファイル検知ごとに短命なスレッドを 1 本生成（`daemon=True`）。ダイアログ表示後は消滅し、スレッドは蓄積しない
- **結論: リソース消費は最小限であり、他アプリのパフォーマンスに影響しない**

### 他アプリの UI への干渉

- Tkinter の `-topmost` 属性により、ダイアログは常に最前面に表示される
- これはユーザーへの通知を確実にするための意図的な設計であり、他アプリの操作自体をブロックしない（モーダルロックなし）
- フルスクリーンアプリ（プレゼン・ゲーム等）使用中でもウィンドウが出現する
- キーボードフック・マウスフック・アクセシビリティ API の使用はなし
- **結論: 他アプリの動作は阻害しない。最前面表示のみ視覚的に出現する**

### ブラウザ（Chrome / Firefox / Safari）とのダウンロード競合

- `.crdownload`（Chrome）/ `.part`（Firefox）/ `.tmp` `.download` `.partial` の一時拡張子ファイルは除外
- `on_moved` イベントで一時ファイルから実ファイルへのリネームを検知し、完了後に処理
- 検知後 0.8 秒の待機でファイル書き込み完了を確認してから UI を表示
- `_pending` セットにより、同一ファイルの二重処理を防止
- ユーザーが UI 操作をする間（数秒以上）、ブラウザの書き込みは必ず完了している
- **結論: ブラウザのダウンロード処理と競合しない**

### macOS のシステム整合性（SIP）

- System Integrity Protection（SIP）保護対象の `/System` / `/usr` / `/bin` 等へのアクセスなし
- 標準の macOS ファイルシステム API のみ使用
- **結論: SIP との競合なし**

### 多重起動の防止

- `kiro.lock`（アプリフォルダ内）に実行中プロセスの PID を書き込む PID ロックファイル方式で多重起動を防止
- 起動時にロックファイルが存在する場合、記載 PID が実行中かを `os.kill(pid, 0)` で確認し、起動中であればエラーメッセージを表示して終了
- ロックファイルが存在しても PID が無効（クラッシュ等による古いロック）の場合は上書きして起動を継続
- 正常終了（Ctrl+C）時はロックファイルを自動削除。`finally` ブロックで保証されるため、例外発生時も確実に削除される
